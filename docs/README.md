# Docs Template — AI 辅助编程项目的文档治理模板

> **职责**：这是一个可直接复制到其他项目的文档治理模板，采用"动态文档（Living Documentation）+ 归档"两层结构。专为 AI 辅助编程（Vibe Coding）场景设计。

> **使用声明（请先看）**：这套模板是思想示例，不是格式规范。你可以完全自由修改内容结构、标题命名、段落写法，**不需要照抄本 example**。唯一建议是保留每个文件的核心职责（例如 `STATUS` 负责状态、`DECISIONS` 负责决策）。

> **强烈鼓励**：直接使用 AI 生成和迭代这些文档。重点不是文档长得是否一致，而是这些文档能否持续承载项目记忆、降低认知负担、帮助 Agent 更快接入编码流程。

---

## 为什么需要这套文档模板？

### 1. AI 时代，文档比以前更重要

无论你是在做 research、building 还是 engineering，项目过程中总会沉淀大量知识——业务决策、口径定义、踩过的坑、约定俗成的规范。

**我们强烈建议用文档把这些知识沉淀下来。** 原因很简单：AI 最强大的能力就是快速迭代，但快速迭代的代价是很容易"忘掉"之前迭代过程中积累的经验。文档是对抗遗忘的唯一可靠手段。

好消息是，AI 时代写文档不再是一件痛苦的事：

- **传统时代**：人写文档又慢又累，所以大家普遍没有写文档的习惯。
- **AI 时代**：AI 写文档又快又好。让 AI 在完成一段工作后顺手把经过和结论记下来，几乎零成本。

### 2. 文档对 Coding Agent 的直接价值

当项目逐渐变大，好的文档能带来以下关键好处：

- **省 token、省时间**：Agent 不需要每次都扫描整个代码库来理解项目——读几个文档就能对项目有清晰全面的认识，大幅减少上下文消耗。
- **避免重复踩坑**：之前上线的功能出了 bug、改掉了，下次遇到类似问题，Agent 直接看记录就能修。不用再从头排查一遍。
- **守住约定俗成的规范**：前端用什么颜色、什么组件；后端怎么命名、怎么分层——这些规范写着写着连人自己都会忘，但 Agent 看了文档就不会乱写。
- **让不同 Agent 会话之间保持连续性**：每轮新对话就像一个"失忆"的新同事上岗，文档就是它的交接手册。

### 3. 按日期写文档的痛点

大多数人写文档的习惯是按日期记录，类似日志：

```
20260113_主题A.md
20260228_主题B.md
20260303_复盘C.md
...
```

这种方式的问题是：**时间一久，你自己不看了，Agent 也不看了。** 之前记录的关键决策、踩过的坑、达成的共识，全都沉没在一堆日期文件里。认知负担越来越重，最终等于没写。

### 4. 解法：6 个动态文档

核心思路是把文档分成两层：

- **第一层：6 个动态文档**——数量少、常更新，是项目知识的唯一入口。
- **第二层：归档（archive）**——日期文件照写不误，但写完就归档，不再作为日常查阅入口。

你每天可以写很多文档，但**每产生一个业务决策、每更新一次项目状态，都要把关键信息同步到这 6 个动态文档里**。同时要注意保持动态文档精简——它们不是百科全书，只记“Agent 和你需要知道的最少必要信息”。

这样，无论是你自己还是 Agent，任何时候想了解项目现状，**只需要读这 6 个文件**。

---

## 6 个动态文档是什么

| 文件 | 回答的核心问题 | 什么时候更新 |
|---|---|---|
| **OVERVIEW.md** | “这个项目是什么？解决什么问题？整体架构是怎样的？” | 项目有重大变动时 |
| **STATUS.md** | “项目现在进展到哪了？什么做完了？什么没做？” | 每完成一个里程碑 |
| **DECISIONS.md** | “我们之前为什么这么决定的？” | 每做一个重要决策 |
| **GLOSSARY.md** | “这个变量/术语/口径到底什么意思？” | 引入新概念时 |
| **RUNBOOK.md** | “怎么跑代码？产物在哪？出错怎么排查？” | 运行方式变更时 |
| **CONVENTIONS.md** | “代码风格和约束规则是什么？” | 规范新增或变更时 |

## 目录结构

```text
docs_template_example/
├── README.md                        ← 你正在看的这个文件（为什么 + 怎么用）
├── OVERVIEW.md                      ← 项目总览与架构
├── STATUS.md                        ← 项目进展与待办
├── DECISIONS.md                     ← 关键决策记录
├── GLOSSARY.md                      ← 术语与变量定义
├── RUNBOOK.md                       ← 运行方法与产物位置
├── CONVENTIONS.md                   ← 编码规范与约束
├── HOW_TO_CLONE_THIS_TEMPLATE.md    ← 迁移到新项目的步骤
└── archive/
    └── README.md                    ← 归档规则说明
```

## 使用方式（迁移到新项目）

1. 复制整个 `docs_template_example/` 到目标项目，改名为 `docs/`
2. 按项目真实情况填充六个动态文档
3. 日常产生的日期文档放入 `archive/`
4. 在项目级 Agent 配置文件（如 `CLAUDE.md`）写入"必读文档"入口

详细步骤见 [HOW_TO_CLONE_THIS_TEMPLATE.md](HOW_TO_CLONE_THIS_TEMPLATE.md)。

## 推荐维护节奏

- 项目重大变动 → 更新 `OVERVIEW.md`
- 里程碑结束 → 更新 `STATUS.md`
- 重要决策产生 → 追加 `DECISIONS.md`
- 引入新变量/术语 → 更新 `GLOSSARY.md`
- 运行方式变更 → 更新 `RUNBOOK.md`
- 规范新增或变更 → 更新 `CONVENTIONS.md`

## 设计原则

- **动态文档是唯一真相源（single source of truth）**——与旧文档冲突时，以动态文档为准
- **归档只回溯，不作为日常入口**——写完就放进 `archive/`，不再修改
- **决策只追加不删除**——推翻旧决策时新增一条标注 supersede，不抹掉历史
- **保持精简**——动态文档膨胀到没人想看，就失去了意义
